##
When developing automated/mechanical trading systems and backtesting and optimizing over historical data, how do I find the balance between shorter training window, so the system can adapt to changing market condition versus longer training window to have a larger sample size?
##
The core challenge is a classic bias-variance / relevance-robustness tradeoff in non-stationary markets. Markets evolve (regime shifts in volatility, trends, liquidity, macro drivers, etc.), so a very long training/optimization window risks including outdated regimes that no longer apply, making your system lag real-time conditions. A short window lets it adapt quickly but often yields noisy, statistically weak results (few trades → high variance, easy overfitting to recent quirks or noise).
The standard, battle-tested way pros resolve this is Walk-Forward Optimization/Analysis (WFO/WFA), pioneered by Bob Pardo and widely recommended in quantitative trading literature (e.g., by Ernie Chan, Marcos López de Prado). It simulates real-world periodic re-optimization while rigorously validating on truly unseen future data multiple times.
How Walk-Forward Optimization Works (and Why It Balances Both Sides)

Split historical data into sequential blocks.
Optimize parameters on an in-sample (IS/training) window (e.g., find best moving-average periods, stop levels, etc.).
Apply those fixed parameters to the next out-of-sample (OOS/testing) window (unseen data) and record performance.
Roll the windows forward by a step size (often = OOS length or a fraction) and repeat.
Aggregate all the OOS periods into one true out-of-sample equity curve and evaluate overall robustness.

This gives:

Adaptation → Parameters are periodically re-optimized on recent data (short effective "memory").
Large effective sample size / robustness → You get many independent OOS tests covering different market conditions, plus statistical confidence from repeated validation.

Two main variants (choose/test both):

Rolling (non-anchored): Fixed-length IS window (always uses only the most recent N periods). Better for fast adaptation when regimes change frequently.
Expanding (anchored): IS window grows over time (starts from the beginning and adds data). Gives more sample size as you go; good if you believe longer-term structure persists.

Practical Guidelines for Choosing Window Lengths
There is no universal "perfect" number — it depends on your strategy's trading frequency, holding period, and market (stocks vs. forex vs. crypto). Experiment with a matrix of configurations (meta-WFO) and pick the one with the best aggregated OOS metrics. Common rules of thumb from practitioners and platforms (QuantInsti, MultiCharts, Build Alpha, etc.):

Number of windows: At least 8–15 OOS periods for confidence (more is better).

How to Evaluate and "Find" the Right Balance for Your System
Don't just pick one window length and declare victory — test robustness explicitly:

Walk-Forward Efficiency (WFE): (OOS performance) / (IS performance). >50–70% is often considered good (shows it didn't degrade too much out-of-sample).
Consistency across OOS periods: Stable Sharpe, profit factor, max DD, win rate. Low variance is key.
Parameter stability: Optimized params shouldn't jump wildly between windows (look for "plateaus" in the parameter space where many nearby values perform similarly — use clustering or a Parameter Stability Index). Sharp single peaks = overfitting.
Other metrics: Aggregated OOS Sharpe/return/drawdown, number of profitable OOS windows (>70–80% ideal), equity curve smoothness.
Robustness stress-tests:
Run the same WFO on different starting dates or sub-periods (regimes).
Perturb parameters slightly and re-test.
Monte Carlo resampling, purged/combinatorial cross-validation (López de Prado style — avoids lookahead and overlapping bias).
Multi-asset or walk-forward on out-of-sample future data if available.


If OOS collapses with shorter windows → lengthen IS (more data). If performance lags market changes or params drift a lot with longer windows → shorten (more adaptation).
Additional Best Practices to Avoid Pitfalls

Always include realistic transaction costs, slippage, and commissions — they amplify overfitting on short windows.
Prioritize recent data but ensure coverage of multiple regimes (bull/bear, high/low vol) in the overall history.
For indicators/features: Use rolling or expanding windows for normalization/lookbacks to prevent lookahead bias.
Don't over-optimize the window lengths themselves (data-snooping risk) — limit to a small grid search.
Regime detection (e.g., HMM, volatility clustering) can make windows dynamic/adaptive as a next-level step.
Software: Many platforms (TradeStation, MultiCharts, QuantConnect, Python with Backtrader/Backtesting.py + custom loops) have built-in WFO.

This approach is far superior to a single fixed long backtest (which hides overfitting) or pure short-window rolling (which lacks validation). In production, the "balance" is dynamic: your system adapts via periodic re-optimization calibrated exactly by the WFO that survived rigorous OOS scrutiny.

##
Yes, this pre-filtering step is useful in principle — but only if implemented without data leakage. It can meaningfully improve robustness by focusing on features that show consistent predictive power (or stability) across multiple validation folds, rather than noisy ones that spike in one regime. In high-dimensional quant feature sets (technicals, macro, sentiment, etc.), this kind of dimensionality reduction + stability check often reduces overfitting, improves out-of-sample Sharpe/drawdown consistency, and speeds up the main training. Practitioners (e.g., in Alpha Scientist workflows, Robot Wealth experiments, or Boruta/RFE pipelines) frequently use similar repeated-CV stability ranking exactly for this reason. The "n repeats" part is smart for variance reduction.
However, standard k-fold (even repeated) on the full historical dataset introduces subtle but real look-ahead bias. Even though you're not training the final model on it, the selected features have been "pre-vetted" using information from the entire period — including future regimes, volatility clusters, or correlations that wouldn't have been known at time t. This is classic selection bias / implicit leakage in financial ML (heavily warned about in López de Prado's Advances in Financial Machine Learning, QuantInsti pipelines, and StackExchange/Reddit quant threads). Your features end up being those that happened to be stable across the whole backtest, which overstates their true forward-looking stability.
For Walk-Forward Tests: Yes — Do the Pre-Filtering Only on Each [t – L, t] IS Window
This is non-negotiable for a clean, realistic simulation. Every step of your pipeline (feature engineering, pre-filter selection, hyperparameter tuning, model fitting) must use only data available at time t. Otherwise the aggregated OOS results are contaminated, and you lose the whole point of walk-forward (true forward simulation of live periodic re-optimization).
Correct way in WFO:

For each rolling/expanding IS window [t – L, t]:
Generate candidate features only from data up to t (no future leaks in lags, rolling stats, etc.).
Run your repeated k-fold pre-filter inside this window only to select/rank stable features.
(Strongly recommended upgrade) Replace standard k-fold with a time-series-aware CV:
TimeSeriesSplit (scikit-learn)
Purged K-Fold + Embargo (López de Prado style — removes any train samples whose labels overlap temporally with test labels; add a 1–5 bar embargo gap)
Or blocked/rolling CV with no shuffling

This gives you truly out-of-sample stability scores within the IS data.

Train your main model (with the selected features) on the full IS window.
Apply fixed parameters + selected features to the OOS period.
Roll forward and repeat.

This makes your feature set adaptive (which is actually a feature, not a bug — markets change, so stable features today may not be tomorrow). You'll see some turnover in the selected features across windows; track that as an extra robustness diagnostic (low turnover = more structural edge; high turnover = regime-dependent, consider economic rationale filters too).
Practical tips to make it work well:

If your IS window is short (<1–2 years or <200–500 trades), repeated CV inside it can still be noisy → consider simpler embedded selection (e.g., Random Forest / XGBoost permutation importance or MDA, computed with purged CV) or hybrid: pre-filter globally with economic/intuitive rationale + statistical checks, then re-rank inside each window.
Alternatives that many pros prefer for trading:
Boruta (all-relevant selection) or recursive feature elimination (RFE) run inside each IS with proper CV.
Tree-based importance + clustering for orthogonal stable features (similar to your current heuristic).
Fixed core features chosen once on very long data (with heavy regime coverage), plus dynamic ones re-selected.

Computational cost: Feature selection inside every window adds time, but it's worth it. Parallelize or cache if needed.
Validation: After full WFO, check WFE ratio, % of profitable OOS periods, parameter/feature stability across windows, and run a "leaky" version side-by-side to quantify how much optimism the global pre-filter added (you'll usually see it inflates results).

def generate_param_batch(n_combos=15000, device="cuda"):

        # active_filters_mask: (N_combos, N_days) bool tensor

            # Vectorized over combos: find short strike index by closest delta

            short_idx = torch.argmin(torch.abs(deltas.unsqueeze(0) - param_batch["short_delta"].unsqueeze(1)), dim=1)
            short_strike = chain[short_idx, strike_col]
            long_strike = short_strike - param_batch["width"]
            
            # Entry mid price
            entry_credit = (chain[short_idx, ask_col] - chain[long_idx, bid_col]) * 100  # multiplier
            
            # Minute-by-minute P&L tracking (vectorized cumsum + SL check)
            spread_value_path = self.compute_spread_value_path(bars, short_strike, long_strike)  # (N_combos, 390) via broadcasting
            pnl_path = spread_value_path * 100 - entry_credit.unsqueeze(1)
            hit_sl = pnl_path <= -param_batch["sl_pct"].unsqueeze(1) * entry_credit.unsqueeze(1)
            exit_pnl = torch.where(hit_sl.any(dim=1), 
                                  pnl_path[torch.arange(batch_size), hit_sl.int().argmax(dim=1)],
                                  pnl_path[:,-1])
            
            daily_pnl = exit_pnl * param_batch["risk_pct"] * account_size  # position sizing
            daily_pnl *= active_filters_mask[:, day_idx]  # zero if filter fails
            equity_curves[:, day_idx] = daily_pnl
        
        # Aggregate metrics per combo
        cum_eq = torch.cumsum(equity_curves, dim=1)
        sortino = self.compute_sortino(cum_eq)  # GPU vectorized
        maxdd = self.compute_maxdd(cum_eq)
        return sortino * (1 - maxdd / 0.20)  # objective

##
A bit of a technical question about thetadata API. I'm looking to use data in 1m intervals. The options quote (and the underlying index quote) API provides the quote in standard market quote, where the given timestamp corresponds to the beginning (open) of the quote. The greeks/implied vol APIs provide data in a snapshot, so the given timestamp corresponds to what the values were at that given time. If we want to avoid look-ahead issues in the backtests than we should combine these based on what data was available at any given point, meaning combine the greeks/implied vol snapshots with the previous minute quote.
What's the best way store these in the Parquet files? Separate table for each API, or a combined table?
##
Separate tables per API endpoint for the raw data. This is the cleanest, most maintainable, and safest approach for your exact constraints (1-minute intervals, different timestamp semantics, strict no-lookahead, and downstream GPU tensor creation).
Why Separate Tables (Not One Combined Table) for Raw Storage

Different native schemas and timestamp meanings
Quotes (and underlying) are interval-based with timestamp = start of the minute (the “open” of that 1-minute bucket).
Greeks / IV are true snapshots with timestamp = exact moment the value was calculated.
Storing them in one table would force you to either duplicate columns, add a lot of nulls, or invent artificial alignment logic at ingestion time — all of which complicates raw ingestion and makes debugging harder.
Incremental fetching & re-processing
You can re-pull or re-process just the greeks/IV for a bad day without touching the massive quote files. ThetaData bulk endpoints are fast but not free; separate storage lets you be surgical.
Parquet columnar efficiency
When you later load only quotes for a window (or only greeks), Parquet predicate pushdown and column pruning are perfect. A single mega-table with 100+ columns wastes I/O and RAM.
Future-proof
If ThetaData adds new endpoints (higher-order greeks, trade greeks, OI snapshots, etc.), you just add another subfolder. No schema migration pain.
No look-ahead risk at storage level
The raw files stay exactly as the API returned them. All alignment logic lives in one place: your preprocessing script (where it belongs and can be versioned in DVC).

How to Align Them Correctly During Preprocessing (The Only Place Alignment Happens)
In your daily preprocessing loop (the same script that builds the Zarr tensors), do a time-as-of merge:
Pythonimport pandas as pd
from pathlib import Path

def build_aligned_daily(date_str: str):
    # Load raw (only this day)
    q = pd.read_parquet(f"data/raw/thetadata/quotes/date={date_str}")
    g = pd.read_parquet(f"data/raw/thetadata/greeks/date={date_str}")
    # underlying if needed...
    
    # Ensure datetime index, floored to minute for quotes
    q['ts'] = pd.to_datetime(q['timestamp'], unit='ms').dt.floor('1min')
    g['ts'] = pd.to_datetime(g['timestamp'], unit='ms')
    
    # Critical: merge greeks onto the PREVIOUS minute's quote
    # (this is the no-lookahead step)
    aligned = pd.merge_asof(
        g.sort_values('ts'),
        q.sort_values('ts'),
        on='ts',
        direction='backward',   # use the quote from the previous minute
        tolerance=pd.Timedelta('59s')  # safety
    )
    
    # Now aligned has: quote columns from t-1min + greeks/IV exactly at t
    # Drop any rows where we couldn't find a prior quote (rare)
    aligned = aligned.dropna(subset=['bid', 'ask'])  # or whatever quote columns you need
    
    # Save the aligned version (optional but very useful)
    aligned.to_parquet(f"data/processed/aligned/date={date_str}.parquet", compression="zstd")
    
    # Then build your GPU tensor directly from this aligned DF
    # (strike-level chain with correct quote + greeks at each snapshot)
    return aligned
This runs in <2 seconds per day and guarantees zero look-ahead: any decision you make at time t only ever sees the quote that was available before the greeks snapshot arrived.
